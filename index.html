Cyber-physical system documentation

An integrated platform for sensing, real-time intelligence, visualization, and human–AI interaction.

CPS Smart Home V2 is a complete cyber-physical system that connects distributed sensors, live data services, interactive dashboards, secure user access, and a locally operated AI assistant. It collects multi-source JSON readings, processes and streams them through a central API, presents system status and analytics in real time, and supports intelligent chat without exposing the local AI service directly to the browser. This guide covers the architecture, installation, configuration, operation, testing, and deployment of the complete platform.

Frontend introduction

The frontend solves the problem of understanding several independent smart-home sensing feeds in one place. It is intended for students, instructors, and laboratory demonstrators. Its scope includes bilingual dashboards, academic-discipline content, authentication integration, sensor visualizations, online/offline device status, event panels, video placeholders, and CPS chat.

Install Node.js, Git, and Yarn

  1. Install Node.js 22 LTS

    Download the installer for your operating system from nodejs.org. Accept the default installer options; npm is included with Node.js. Close and reopen the terminal after installation.

  2. Install Git

    Download Git from git-scm.com and accept the default options. Windows users can then use PowerShell or Git Bash.

  3. Verify Node.js, npm, and Git
    node --version
npm --version
git --version

    Node.js should report version 22.x.

  4. Install Yarn
    npm install --global yarn
yarn --version

    Restart the terminal if the yarn command is not immediately available.

Installation

  1. Download the frontend

    Clone the repository with Git:

    git clone https://github.com/Zamidmustefa/cps_smart_home_frontend_v2.git
cd cps_smart_home_frontend_v2

    Without Git, open the frontend repository, select Code → Download ZIP, extract it, and open a terminal in the extracted folder.

  2. Install all frontend packages

    The frozen lockfile installs the tested dependency versions.

    yarn install --frozen-lockfile
  3. Create the frontend environment file
    # Windows PowerShell
Copy-Item .env.example .env

# macOS or Linux
cp .env.example .env

    Edit .env in a text editor and replace the placeholder Supabase values. Use the local backend URLs shown below when running both projects on one computer.

Backend introduction

The backend provides one stable public gateway for smart-home data and local AI chat. It is intended for student sensor teams, the Vue dashboard, and a computer running Ollama. Its scope includes health endpoints, JSON ingestion, bounded in-memory channel history, SSE updates, optional API-key protection, voice-service proxying, optional Kokoro TTS, and browser-to-agent WebSocket relay.

Leaving DATA_CHANNEL_API_KEY empty is convenient for local development only. Set a strong value in production. The complete .env.example also includes optional Ollama agent, search, voice service, history, and Kokoro settings.

Do not commit secrets. Keep the real .env, data-channel API key, Supabase private credentials, and agent token outside documentation and source control. If a key has already been shared publicly, rotate it.

Smart-home data channels

Each channel is independent. POST accepts a JSON object or array and returns 202 Accepted. Readings are stored in memory and are cleared when the process restarts.

POST/api/data-channels/:channelAccept one new JSON reading.
GET/api/data-channels/:channel/latestReturn the newest reading.
GET/api/data-channels/:channel/history?limit=20Return oldest-to-newest history.
SSE/api/data-channels/:channel/streamPush each successful reading immediately.

Sample request

curl -X POST http://localhost:7020/api/data-channels/body-vitals \
  -H "Content-Type: application/json" \
  -H "X-API-Key: YOUR_INGESTION_KEY" \
  -H "X-Source-Id: student-laptop-01" \
  -d '{"timestamp":"2026-07-19T12:00:00Z","heart_rate_bpm":78}'

Student file senders

The ready-made scripts read a JSON file and send it every three seconds. Update their channel and source identifier before use.

# Node.js 20+
node examples/send-json-file.js data.json

# Python 3; no third-party package required
python examples/send_json_file.py data.json

X-Source-Id is not a password. It is a readable label such as team-2-laptop that identifies the sender in the dashboard.

Demonstration checklist

  1. Start Backend V2 and confirm /health returns an OK response.
  2. Start Frontend V2 and open the Smart Home dashboard.
  3. Run one Node or Python sender for a selected channel.
  4. Confirm the channel becomes Online and its values update.
  5. Stop the sender; confirm the channel becomes Offline after approximately 10 seconds.
Known limitations. Data-channel history is memory-only and is lost on restart. Large frontend bundles currently produce a Vite chunk-size warning. The local Ollama agent must be online for CPS chat, and Kokoro TTS is intentionally disabled on memory-constrained cloud instances.

Troubleshooting

The most common setup and runtime problems.

Frontend loads but live data does not appear

Confirm VITE_API_URL points to the running backend, restart Vite after editing .env, and open /api/data-channels directly to verify connectivity. Also confirm the backend CORS list includes the frontend origin.

A student sender receives HTTP 404 “Route not found”

Use the exact path /api/data-channels/CHANNEL_SLUG. Valid slugs are csi-wifi, body-vitals, lighting-acoustic, hic, and gaze-vision. Redeploy the backend if the cloud service still runs an older revision.

A student sender receives HTTP 401 or 403

The sender key does not match the backend DATA_CHANNEL_API_KEY. Send it as X-API-Key or Authorization: Bearer …. Never place this key in frontend code.

CPS chat says the local Ollama computer is offline

Start Ollama, then start local-agent.js. Check that CLOUD_WS_URL uses ws:// locally or wss:// in production and that the local and cloud AGENT_TOKEN values match.

The cloud instance exceeds 512 MB of memory

Keep KOKORO_ENABLED=false. The ONNX speech model can exceed a small instance limit. Also retain the per-channel byte cap, reduce large CSI payloads, and keep channel history short.
